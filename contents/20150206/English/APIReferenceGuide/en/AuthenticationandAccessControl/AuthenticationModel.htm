Authentication Model

The PCS authentication is based on OAuth 2.0 framework. OAuth (Open Authorization) is an open protocol that allows secure API authorization in a simple and standardized way. The OAuth allows you to access resources of some other Cell for a limited period of time. This limited period becomes invalid after the defined period of time.

Accessing the PCS portal is not only based on the traditional client-server authentication model, but also based on the authentication model where the application issues tokens. These tokens are used to access data through various cells.

The PCS portal is supporting the following authorization flows:

·        Resource Owner Password Credentials (ROPC) flow

·        Implicit flow (not supported in Version 1.0)

The PCS portal consists of the following two authentication frameworks:

·        User authentication

·        Application authentication

When you get an authentication from one cell, you can get the authentication of another cell. This makes the social user authentication operation possible.

  • User Authentication

User authentication consists of:

Account authentication

In account authentication, when an account is created in a cell with an account name and password, it also gets a token.

Basic authentication

In basic authentication, you can access resources simply using your ID and password. You do not need token authentication.

Note:

This feature is not available in PCS Version 1.0.

Trancell user authentication

In Transcell user authentication, when an account is created in a cell with an account name and password, it also gets a token. Further, it also generates a transcell token, which helps in accessing other cell.

  • Schema Authentication

PCS authenticates each application by its schema, known as schema authentication. Application cell (schema cell) corresponds authentication using ID and password in the Application.

For Persona container application development, you can use schema authentication instead of implicit flow based authentication to ensure the security.

  • Types of Tokens

There are the following types of tokens:

·        Access token

·        Refresh token

Access Token

An Access Token refers to the credentials used to access protected resources. In case of PCS portal, you need an access token to access the particular Cell. When the access token expires, you must obtain a new access token via a refresh token.

It is a string representing an authorization issued to the particular user (account). The Access Token is same for all the cells created under one environment.

The PCS portal supports two types of Access Tokens: Local cell token and Trans cell token.

Local cell token

The local cell token refers to the token issued for the particular cell. This token is valid only to access the resource of the authenticated cell.

The local cell token expires in 1 hour from the time of issue.

Transcell token

The transcell token is used to access authentication as well as other cells. This token is only valid for the Cell that is specified in the URL of the dc_target parameter.

To make other cells use the transcell token, you must configure the cell control objects, such as ExtCell, Relation, and ExtRole.

In addition, the x509 certificate authentication mechanism is incorporated in the transcell token that contains the CS-approved certificate.

The transcell token expires in 1 hour from the time of issue.

Refresh Token

Refresh token is issued along with the access token.

Refresh token refers to reissue the specified access token. An access token is valid for 1 hour from the time of issue. In order to keep using the PCS portal for a long time, the refresh token is keeps track of the time and as soon as the time is about to over it sends an issue request for the new access token. The refresh token can be reissued even though the access token is not expired. The refresh token expires in 24 hours from the time of issue.

  • Access resources of Cells

This section explains how you can authenticate and access Cell(s) resources. You can use the following methods to access the resources:

·        Access token

·        Refresh token

Access Token

You need Access token to perform operations, such as create cells, update cell, create box, and create account. The Access token is valid for one hour.

For example, to access resources of Cell A, the following process is used:

Figure 1: Access Resources using access token

  1. Account (password) authentication for Cell A. The account is authenticated by passing the correct values in the following parameters of a curl command:

grant_type=password&username=[user name]&password=[password]

  1. You will get the access token from the response body.

  2. Set the access token at authorization header to access the resources.

Authorization: Bearer [access token]

You can access resources using any of the following method:

·        PCS portal

·        CURL Command (when using the setup without the PCS portal)

PCS Portal

To access the resources using PCS portal:

  1. Login to the portal with valid Username and Password. If you enter the correct credentials, the Home page appears. This page contains a list of Unit URLs along with the respective environments appearing beneath every Unit URL.

  2. Click the specified environment. When you click the respective environment, the Access token is generated. To view the value of the Access token, click ‘Show Access Token’ icon appearing in the header.

You can use the Access token to perform operations, such as create cell, update account, create role, and delete external role.

CURL command

Use the following CURL command to authenticate the user (account) and generate the access token:

curl -X POST https://fqdn/__auth\

 -d "dc_target=[unit URL]/&dc_env=[environment ID]&grant_type=password&username=[user name]&password=[password]"\

 -i -k –v

 

where:

dc_target: contains the Unit URL

dc_env: contains the Environment ID

 

The access token is generated, which you can use to perform operations. You can get the value of access token from the “"access_token":”parameter of the response rendered by the API.

You should set the value of access token in the header (-H) part of the command.

Authorization: Bearer token

The following sample CURL command shows how to pass the Access token when you are creating a cell:

curl -X POST "https://fj.baas.jp.fujitsu.com/__ctl/Cell" \

 -d "{\"Name\":\"SampleCellName\"}" \

 -H "Accept:application/json" \

 -H "Authorization:Bearer TOKEN” –i –k -v

Refresh Token

The Web server OAuth authentication flow provides a refresh token that can be used to obtain a new access token.

Access tokens have a limited lifetime of one hour, which is specified by the session time-out in the PCS portal. If an application uses an expired access token, an “Expired access token” error is returned.

The process to access resources of Cell A is as follows:

Figure 2: Access Resources using access token

  1. Account (password) authentication for Cell A. The account is authenticated by passing the correct values in the following parameters of a curl command:

grant_type=password&username=[user name]&password=[password]&dc_target=[Unit URL]

  1. You will get the refresh token from the response body.

  2. Refresh the authentication result of Cell A

grant_type=refresh_token&refresh_token=[Refresh token above]

  1. Get the access token from response body of 3.

  2. Set the access token received from 4 at authorization header to access the resources.

Authorization: Bearer [access token]

You need Refresh token to renew the Access token.

To access resources using PCS portal:

  1. Login to the portal with valid Username and Password. If you enter the correct credentials, the Home page appears. This page contains a list of Unit URLs along with the respective environments appearing beneath every Unit URL.

  2. Click the specified environment. When you click the respective environment, the Access token is generated. It also generates the Refresh token.

The Refresh token is used to obtain a new access token and is valid for 24 hours.

  • Unit User

Refers to the owner of the particular Unit. The Unit user has a right to perform CRUD operations, such as create, read, update, and delete cell(s) and should be able to access unit level API operations. The unit user should have unit user token that helps to access API operations.

Each cell has a hidden attribute called owner, which sets the unit user name as the cell owner at the time of creating the cell.

To perform the CRUD operations on a cell, the attributes of owner and unit user should match.

Unit User Role

The unit user role helps in accessing the PCS portal. The PCS portal recognizes the following as the role of the unit user:

{issuerUrl}/__role/__/unitAdmin Unit controller

 

The unit user role allows the unit user to perform processes related to the particular unit.
